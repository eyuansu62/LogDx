This file is the planning surface for the v2 corpus expansion (E10). It
records target counts per axis (split, failure category, ecosystem, log
size, signal position, evidence format) **before** cases are collected,
so that selection bias from "what was easy to find" can be detected and
corrected during collection.

The target counts here are **not strict quotas**. They are the shape we
are aiming for. A row in the "actual" column that is more than 1 below
the target is a deficit to fix before freezing v2; a row more than 2
above target should be re-examined for over-representation.

## 5. Log-size targets

Buckets and edges are unchanged from `case_tags.schema.json`:

```text
small:   < 500 lines
medium:  500 – 5,000 lines
large:   5,000 – 50,000 lines
huge:    > 50,000 lines
```

| Bucket  | v1.3 actual | v2 target | New needed |
|---------|---:|---:|---:|
| small   | 5  | 12 | 7  |
| medium  | 7  | 18 | 11 |
| large   | 4  | 15 | 11 |
| huge    | 0  | 5  | 5  |

The `huge` row is the highest-priority gap. v1.3 has no log over ~50k
lines, so `tail-200`'s relative strength on short-log-heavy splits is
distribution-confounded. The E10 report cannot make claims about how
hybrid behaves on huge logs unless ≥ 5 huge cases are present.

## 6. Signal-position targets

```text
early:      first 25% of the log contains the critical signal
middle:     25–75%
late:       last 25%
scattered:  critical signal spans > 30% of the log (e.g. matrix logs
            with one failing leg buried inside many passing ones)
```

| Position   | v1.3 actual | v2 target | New needed |
|------------|---:|---:|---:|
| early      | 1  | 8  | 7  |
| middle     | 4  | 12 | 8  |
| late       | 9  | 15 | 6  |
| scattered  | 2  | 10 | 8  |

v1.3 is heavy on `late` because GitHub Actions logs naturally end with
the failing step; this is exactly what `tail-200` is good at. v2 must
add enough `early` and `scattered` cases to test whether `tail-200`'s
v1.3 strength was a real strength or a distribution artifact.

`scattered` is also the failure mode hybrid is most exposed to: when
the relevant signal is split across two logically-distinct sections,
the 4k window can drop one of them.

## 9. Required schema extensions

Before the new cases below can be tagged, `schemas/case_tags.schema.json`
needs minimal extensions. None of these change semantics for v1.3.

```text
failure_category enum: add
  matrix_or_monorepo_failure

evidence_formats enum: add
  assertion_diff
  snapshot_diff
  docker_build_output
  package_manager_error
  timeout_marker
  oom_marker
  matrix_summary

new top-level fields (optional, default-null):
  origin:           enum { legacy_v1_3, new_v2 }
  ecosystem:        enum { python-pytest, python-poetry-pip-tox,
                           javascript-jest, javascript-pnpm-yarn-npm,
                           typescript-tsc, rust-cargo, go,
                           java-maven, java-gradle,
                           docker-buildkit, terraform, kubernetes-helm,
                           c-cpp-cmake, ruby-bundler,
                           generic-github-actions, other }
  primary_language: free string  (already present in tags.json examples)
  ci_provider:      enum { github_actions, gitlab_ci, circleci,
                           buildkite, jenkins, other }
                    (default github_actions for v1.3 + most of v2)
  repo_visibility:  enum { public, private, redacted }
```

The schema bump should be **additive** — every v1.3 `tags.json` must
remain valid. Default-null on the new fields makes that automatic; the
`origin` field gets backfilled to `legacy_v1_3` for v1.3 cases via a
small migration in `tools/tag_cases.py`.

## 10. Anti-bias rules during collection

These rules are encoded in `cilogbench_v2_collection_guidelines.md`.
Repeated here because they directly shape the matrix:

1. Do **not** select cases by trying methods on them first. Cases are
   chosen from CI failures the collector encountered organically, then
   imported, then tagged, then evaluated. Reverse order leaks.
2. Do **not** refuse a case because it looks "too easy" or "too
   adversarial" for hybrid. Both are valid corpus members.
3. Do **not** balance the matrix by editing logs. If a category is
   short, find another natural failure in that category.
4. The matrix is descriptive of what we have, not prescriptive of
   what must exist before freezing — so under-represented cells are
   recorded as caveats in the v2 report rather than papered over.

## 11. Open questions

These are flagged here so they don't drift into the report unannounced.

1. **`matrix_or_monorepo_failure` vs `multi_failure`.** A matrix run
   with one leg failing is not the same as a single-job log with two
   independent failures. The proposed enum split keeps them separate;
   if a case is genuinely both, it gets `multi_failure: true` and
   `failure_category: matrix_or_monorepo_failure`.
2. **`unknown` ecosystem.** Some failures (e.g. raw `bash` step
   crashes) don't fit the ecosystem list. Use `generic-github-actions`
   unless a more specific ecosystem applies; do not add a `null`
   ecosystem.
3. **Stress vs holdout for new cases.** Default new cases to
   `v2/holdout`. Promote to `v2/stress` only if the case is explicitly
   probing a known weakness (per
   `docs/corpus/stress_case_selection.md` style). Reserve `v2/dev`
   for cases that are useful as smoke tests.
